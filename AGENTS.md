# Platzhalter — Agent Instructions

> Instructions for AI coding agents (Claude, Copilot, Cursor, etc.) working on this codebase.
> Read this file before making any changes.

## Project Context

**Platzhalter** is a seating plan management platform for event organizers. It lets a small team (2–3 admins/moderators) design floor plans, assign guests to seats, organize groups, and export seating lists as PDF. There is no public/guest access.

See `DESIGN.md` for full architecture, tech stack, and branding details.

## Development Environment

- **Always use `devbox shell`** to enter the development environment. Never install global packages.
- **Start all services**: `devbox services up` (starts postgres, mailpit, backend, frontend)
- **Package manager**: Use `bun`, not `npm` or `npx`. Use `bunx` instead of `npx`.
- **Environment variables**: Defined in `.env` (not committed). Copy `.env.example` to `.env` for a fresh setup.
- Services:
  - Frontend: `http://localhost:3000`
  - Backend API: `http://localhost:8080`
  - Mailpit UI: `http://localhost:8025`
  - PostgreSQL: `localhost:5432` (database: `platzhalter`)

## Backend Rules (`/backend`)

### Structure
- Go code lives in `/backend`, follows standard Go project layout
- Entrypoint: `cmd/server/main.go`
- Business logic: `internal/` (config, db, handler, middleware, model)
- Migrations: `migrations/` (golang-migrate format: `NNNNNN_name.up.sql` / `.down.sql`)
- SQL queries: `sqlc/query.sql` → generated code in `internal/db/queries/`

### Database
- **Never use an ORM.** Use sqlc for all database queries.
- Write annotated SQL queries in `sqlc/query.sql`
- Write schema changes as migrations in `migrations/`
- After adding/changing SQL: run `sqlc generate` from `/backend`
- Run migrations: `migrate -path ./backend/migrations -database $DATABASE_URL up`

### API
- All endpoints under `/api/v1/`
- JSON request/response bodies
- Semantic HTTP status codes
- Error responses: `{ "error": "message", "code": "ERROR_CODE", "request_id": "..." }`
- Idempotent write operations, optimistic locking where needed

### Configuration
- 12-factor: all config via environment variables
- Loaded by `caarlos0/env` with `godotenv` fallback to `../.env`
- Never hardcode secrets, URLs, or ports

### Logging
- Use `zerolog` for structured logging
- Include request IDs for tracing

### Hot Reload
- Backend uses `air` for hot reload — don't restart manually
- Config in `/backend/.air.toml`

## Frontend Rules (`/frontend`)

### Structure
- Next.js App Router in `/frontend`, TypeScript, Tailwind CSS v4
- Pages in `src/app/`
- Components in `src/components/`
- Utilities in `src/lib/`
- Internationalization in `src/i18n/` and `src/messages/`

### UI Components
- **Use shadcn/ui components** — don't create custom UI primitives
- Components are in `src/components/ui/` (auto-generated by shadcn CLI)
- To add a new component: `bunx --bun shadcn@latest add <component-name>`

### Internationalization
- i18n via `next-intl`
- Default locale: `de` (German), secondary: `en` (English)
- All user-facing strings in `src/messages/de.json` and `src/messages/en.json`
- Never hardcode user-facing text in components

### API Communication
- API calls go through `src/lib/api.ts` wrapper
- Frontend proxies `/api/*` to Go backend via Next.js rewrites (see `next.config.ts`)
- Never create Next.js API routes for business logic — all business logic is in the Go backend

### Auth
- BetterAuth client setup in `src/lib/auth.ts`
- JWT stored in HttpOnly cookies, **never in localStorage**
- Use `useSession()` hook for auth state

### Canvas & Drag-and-Drop
- Floor plan editor uses `react-konva` (Konva.js)
- Drag-and-drop uses `@dnd-kit/core` and `@dnd-kit/sortable`
- PDF generation uses `@react-pdf/renderer`

### UX Requirements
- **Mobile-first design** — design for touch screens (>= 360px) first
- All interactive elements: touch targets >= 44x44pt
- Critical actions (seat assignment, edge tap): >= 56x56pt
- No hover-only interactions — everything must work via tap, long-press, or swipe
- Use Shadcn Drawer (Sheet) for context menus on mobile (long-press trigger)
- Toast notifications via Sonner (not modal alerts)

## Roles

| Role      | Can do                                                              | Cannot do              |
|-----------|---------------------------------------------------------------------|------------------------|
| Admin     | Everything: users, floor plans, tables, seats, groups, PDF export   | —                      |
| Moderator | Tables, seats, groups, drag-and-drop, PDF export                    | User management        |

There is no guest/viewer/public role.

## Do NOT

- Commit `.env` files (they contain secrets)
- Use `npm` or `npx` (use `bun` / `bunx`)
- Add ORM dependencies (use sqlc)
- Create API routes in Next.js for business logic
- Store JWTs in localStorage (use HttpOnly cookies)
- Add hover-only interactions (must work on touch)
- Install global packages outside devbox
- Edit files in `internal/db/queries/` (auto-generated by sqlc)
- Skip migrations — always write proper up/down SQL files

## Common Commands

```bash
# Enter dev environment
devbox shell

# Start all services (postgres, mailpit, backend, frontend)
devbox services up

# Add a Go dependency
cd backend && go get <package> && go mod tidy

# Add a frontend dependency
cd frontend && bun add <package>

# Add a shadcn component
cd frontend && bunx --bun shadcn@latest add <component>

# Create a new migration
migrate create -ext sql -dir backend/migrations -seq <name>

# Run migrations
migrate -path ./backend/migrations -database $DATABASE_URL up

# Generate sqlc code
cd backend && sqlc generate

# Build backend manually
cd backend && go build -o tmp/server ./cmd/server
```
