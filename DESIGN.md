# Platzhalter — Design Document

## Overview

Platzhalter is a seating plan management platform for event organizers. It enables 2–3 admins/moderators to collaboratively design floor plans, assign guests to seats, manage groups, and export seating lists as PDF. There is no public-facing or guest-access role — the platform is exclusively for internal planning.

## Architecture

```
┌──────────────┐     REST/JSON      ┌──────────────┐     SQL      ┌────────────┐
│   Next.js    │ ←── Bearer JWT ──→ │   Go Fiber   │ ←── pgx ──→ │ PostgreSQL │
│  (Frontend)  │                    │  (Backend)   │              │            │
│  App Router  │                    │  /api/v1/*   │              │            │
└──────────────┘                    └──────────────┘              └────────────┘
       ↑                                   ↑
  BetterAuth                          JWT Validation
  (Session + JWT)                     (Middleware)
```

- **Frontend (Next.js)**: Pure UI layer. No business-logic API routes. Proxies `/api/*` requests to the Go backend via Next.js rewrites.
- **Backend (Go Fiber)**: Handles all business logic, database access, and authorization. Stateless REST API.
- **Database (PostgreSQL)**: Schema managed by golang-migrate. Queries generated by sqlc — no ORM.
- **Auth (BetterAuth)**: Lives on the Next.js layer. Issues JWTs with role and user ID claims. Go Fiber validates tokens via middleware.

## Roles & Permissions

| Role      | Capabilities                                                                 |
|-----------|-----------------------------------------------------------------------------|
| Admin     | Full access: users, floor plans, tables, seats, groups, PDF export          |
| Moderator | Tables, seats, groups, drag-and-drop, PDF export. No user management.       |

There is no guest/viewer role. Invitations are email-based with time-limited links via BetterAuth.

## Auth Flow

1. Admin invites user via email (BetterAuth invite flow)
2. User registers and logs in → BetterAuth creates session
3. BetterAuth issues JWT (access: 15min, refresh: 7d) with claims: `{ user_id, role }`
4. Frontend sends JWT in `Authorization: Bearer <token>` header
5. Go Fiber JWT middleware validates signature, expiry, issuer
6. Role extracted from claims — no DB query per request
7. Silent refresh in Next.js client

## Data Flow

```
User Action → Next.js UI → fetch('/api/...') → Next.js Rewrite Proxy
  → Go Fiber → JWT Middleware → Route Handler → sqlc Query → PostgreSQL
  → JSON Response → Next.js UI
```

## Key Modules

### Floor Plan Editor (Konva Canvas)
- Top-down view of tables, shapes, and seats
- Draw tools: rectangle, circle, line, polygon
- Table objects with per-edge seat assignment (tap edge → enter count)
- Snap-to-grid and table-to-table magnetic snapping
- Undo/redo (20+ steps), pinch-to-zoom, one-finger pan
- State persisted as JSON in PostgreSQL

### Seating Assignment
- Tap a free seat → enter name → seat is occupied
- Visual distinction: free / occupied / parked
- Real-time sync between sessions (SSE or polling)

### Groups & Drag-and-Drop
- Named groups with unique colors
- Drag individual or entire group to target seat/table
- Occupied-to-occupied drop = automatic swap
- Ghost preview during drag, collision detection
- Parking area for temporarily unassigned people
- Group constraint: all members must sit at same table

### PDF Export
- Table of all placed persons: Name, Table Nr., Seat Nr., Group
- Sortable by table, name, or group
- Header with event name and date, page numbers
- Optional: graphical floor plan miniature

## Database Conventions

- UUIDs as primary keys (`gen_random_uuid()`)
- `created_at` / `updated_at` timestamps on all tables
- `version` column for optimistic locking on write-heavy tables
- Floor plan layout stored as JSONB
- Migrations in `/backend/migrations/` (golang-migrate format)

## API Conventions

- All endpoints under `/api/v1/`
- JSON request/response bodies
- Semantic HTTP status codes
- Error format: `{ "error": "message", "code": "ERROR_CODE", "request_id": "..." }`
- Idempotent write operations where applicable
- Structured logging with request IDs for tracing

## Environment & Tooling

- **devbox**: Reproducible dev environment (bun, go, air, postgresql, mailpit, sqlc, go-migrate)
- **process-compose**: Orchestrates postgres, migrations, mailpit, backend, frontend
- **air**: Go hot-reload during development
- **bun**: JavaScript runtime and package manager (not npm)
